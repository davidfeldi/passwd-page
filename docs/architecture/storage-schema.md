# Storage Design

## Go Interface

All storage backends implement the `Store` interface. This is the only contract between the application layer and persistence.

```go
package store

import (
    "context"
    "time"
)

type Secret struct {
    ID            string
    Ciphertext    []byte
    BurnAfterRead bool
    ExpiresAt     time.Time
    CreatedAt     time.Time
    Views         int
}

type CreateParams struct {
    Ciphertext    []byte
    BurnAfterRead bool
    ExpiresAt     time.Time
}

type Store interface {
    // Create stores a new secret and returns its generated ID.
    // The ID is generated by the store implementation (16 random bytes, hex-encoded).
    Create(ctx context.Context, params CreateParams) (id string, err error)

    // Get retrieves a secret by ID.
    // Returns (nil, nil) if the secret does not exist or has expired.
    // If the secret has burnAfterRead=true, the implementation MUST atomically
    // delete the secret and return it in the same operation (get-and-delete).
    // Increments the view counter for non-burn secrets.
    Get(ctx context.Context, id string) (*Secret, error)

    // Delete removes a secret by ID. No error if it doesn't exist.
    Delete(ctx context.Context, id string) error

    // Cleanup removes all expired secrets. Called periodically by the cleanup goroutine.
    // Returns the number of secrets removed.
    Cleanup(ctx context.Context) (removed int, err error)

    // Close releases all resources (DB connections, etc).
    Close() error
}
```

### Interface Design Decisions

- **`Get` handles burn-after-read atomically**: The store must ensure that a burn-after-read secret can only be read once, even under concurrent requests. For SQLite, this is achieved via a single transaction with DELETE ... RETURNING. For other backends, this may use optimistic locking or atomic operations.
- **`Get` returns `(nil, nil)` for missing secrets**: This simplifies the handler — no sentinel errors. The handler checks `if secret == nil` and returns 404.
- **`Create` generates the ID internally**: The store owns ID generation to ensure cryptographic randomness and avoid caller mistakes.
- **`Cleanup` returns a count**: Useful for logging and monitoring without exposing internal details.

## SQLite Schema

```sql
CREATE TABLE IF NOT EXISTS secrets (
    id              TEXT PRIMARY KEY,
    ciphertext      BLOB NOT NULL,
    burn_after_read BOOLEAN NOT NULL DEFAULT FALSE,
    expires_at      DATETIME NOT NULL,
    created_at      DATETIME NOT NULL DEFAULT (strftime('%Y-%m-%dT%H:%M:%SZ', 'now')),
    views           INTEGER NOT NULL DEFAULT 0
);

-- Index for cleanup goroutine: find expired secrets efficiently
CREATE INDEX IF NOT EXISTS idx_secrets_expires_at ON secrets(expires_at);
```

### Column Details

| Column          | Type     | Description                                     |
|-----------------|----------|-------------------------------------------------|
| id              | TEXT     | 32-char hex string (16 random bytes)            |
| ciphertext      | BLOB     | Encrypted payload (IV + ciphertext + GCM tag)   |
| burn_after_read | BOOLEAN  | If true, delete after first read                |
| expires_at      | DATETIME | UTC timestamp when secret expires               |
| created_at      | DATETIME | UTC timestamp when secret was created           |
| views           | INTEGER  | Number of times the secret has been read        |

### Key Queries

**Create:**
```sql
INSERT INTO secrets (id, ciphertext, burn_after_read, expires_at, created_at)
VALUES (?, ?, ?, ?, ?);
```

**Get (burn-after-read):**
```sql
DELETE FROM secrets
WHERE id = ? AND burn_after_read = TRUE AND expires_at > ?
RETURNING id, ciphertext, burn_after_read, expires_at, created_at, views;
```

**Get (normal):**
```sql
UPDATE secrets SET views = views + 1
WHERE id = ? AND burn_after_read = FALSE AND expires_at > ?
RETURNING id, ciphertext, burn_after_read, expires_at, created_at, views;
```

Note: Both queries filter on `expires_at > NOW()` to ensure expired secrets are never returned even if cleanup has not yet run. The `RETURNING` clause (SQLite 3.35+) makes the get-and-delete/get-and-update atomic within a single statement.

**Delete:**
```sql
DELETE FROM secrets WHERE id = ?;
```

**Cleanup:**
```sql
DELETE FROM secrets WHERE expires_at <= ?;
```

## SQLite Configuration

```go
// Recommended PRAGMA settings applied at connection time
PRAGMA journal_mode = WAL;       -- Write-Ahead Logging for concurrent reads
PRAGMA busy_timeout = 5000;      -- Wait up to 5s on lock contention
PRAGMA synchronous = NORMAL;     -- Safe with WAL mode
PRAGMA foreign_keys = ON;        -- Enforce FK constraints (future-proofing)
PRAGMA cache_size = -2000;       -- 2MB page cache
```

### WAL Mode Rationale
WAL mode allows concurrent readers and a single writer. This is ideal for passwd.page where reads (secret retrieval) vastly outnumber writes (secret creation). The cleanup goroutine's DELETE operations will not block readers.

### Driver
Use `github.com/mattn/go-sqlite3` (CGo) or `modernc.org/sqlite` (pure Go). The pure Go option is preferred for simpler cross-compilation and single-binary deployment.

## Cleanup Goroutine

A background goroutine periodically removes expired secrets from storage.

```go
func StartCleanup(ctx context.Context, store Store, interval time.Duration, logger *slog.Logger) {
    ticker := time.NewTicker(interval)
    defer ticker.Stop()

    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            removed, err := store.Cleanup(ctx)
            if err != nil {
                logger.Error("cleanup failed", "error", err)
                continue
            }
            if removed > 0 {
                logger.Info("cleanup completed", "removed", removed)
            }
        }
    }
}
```

### Configuration

| Parameter          | Default | Description                          |
|--------------------|---------|--------------------------------------|
| Cleanup interval   | 1 min   | How often the goroutine runs         |
| Context            | app ctx | Cancelled on graceful shutdown       |

### Behavior
- Runs every 60 seconds by default
- Logs only when secrets are actually removed (avoids log noise)
- Gracefully stops when application context is cancelled
- Errors are logged but do not crash the application
- The cleanup goroutine is a safety net; queries also filter on `expires_at`

## Future Pluggability

The `Store` interface is designed for easy addition of new backends:

### Redis
- Use `SET` with `EX` (TTL) for automatic expiration — no cleanup goroutine needed
- Use `GETDEL` for atomic burn-after-read
- Ciphertext stored as byte value; metadata in a hash or encoded in the value
- Connection via `github.com/redis/go-redis/v9`

### PostgreSQL
- Same schema as SQLite, `BYTEA` instead of `BLOB`
- `DELETE ... RETURNING` works natively
- Use `pg_cron` or the same goroutine approach for cleanup
- Connection via `github.com/jackc/pgx/v5`

### Backend Selection

Backend is selected via configuration (environment variable or flag):

```
PASSWD_STORE=sqlite      # default
PASSWD_STORE=redis
PASSWD_STORE=postgres
```

A factory function returns the appropriate `Store` implementation:

```go
func NewStore(backend string, dsn string) (Store, error) {
    switch backend {
    case "sqlite":
        return NewSQLiteStore(dsn)
    case "redis":
        return NewRedisStore(dsn)
    case "postgres":
        return NewPostgresStore(dsn)
    default:
        return nil, fmt.Errorf("unknown store backend: %s", backend)
    }
}
```

### Data Directory

SQLite database file location:
- Default: `./data/passwd.db`
- Configurable via `PASSWD_DB_PATH` environment variable
- Directory is created automatically if it does not exist
- File permissions: `0600` (owner read/write only)
